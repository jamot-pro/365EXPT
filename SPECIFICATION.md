# 365EXPT Repository
Version: 0.1

> This is the founding specification that drives this repository.
> Per our own rules — documentation before implementation — it lives
> here in public, and evolves by pull request like everything else.

---

# Vision

365EXPT is a global, open-source experiment exploring one question:

> Can Artificial Intelligence create more meaningful work than it replaces?

We do not claim to know the answer.

We believe the answer should emerge through transparent experimentation.

This repository is the public laboratory where the experiment happens.

Every discussion, every prototype, every experiment and every discovery lives here.

The repository is designed to become the easiest place in the world for someone to contribute to the future of human-AI collaboration.

---

# Mission

Build a global community that discovers new forms of meaningful work enabled by AI.

We believe AI's greatest contribution is not replacing humans.

It is orchestrating human potential.

---

# Success Criteria

The repository succeeds if:

- Thousands of contributors understand the mission within five minutes.
- Every visitor knows exactly how they can contribute today.
- New contributors can make their first contribution in less than one hour.
- Every experiment produces public knowledge.
- Every improvement strengthens the community.

The repository is not measured by stars.

It is measured by participation.

---

# Repository Principles

1. Everything is public.
2. Everything is experimental.
3. Everything can evolve.
4. People come before technology.
5. AI assists. Humans decide.

---

# Primary User

A student. Age 18-30. Curious about AI. Wants to build something
meaningful. Looking for community. Looking for experience. Looking for
purpose. Not necessarily an experienced developer.

---

# Repository Objectives

The repository should answer four questions:

1. Why does this experiment exist?
2. What are we building?
3. How can I contribute today?
4. What has the community already discovered?

If these four questions are not immediately obvious, the repository has
failed.

---

# Information Architecture

- `README.md` — The public homepage. Explains the experiment. Contains
  one primary Call To Action.
- `START-HERE.md` — A five-minute onboarding. Designed for first-time
  contributors. Should eliminate confusion.
- `CONTRIBUTING.md` — Explains exactly how anyone can participate.
  Includes both technical and non-technical contributions.
- `DISCOVERIES/` — Every important learning. Failures included.
  Knowledge is more valuable than perfection.
- `EXPERIMENTS/` — Each experiment receives its own folder. Contains:
  Problem, Hypothesis, Method, Results, Learned Lessons, Next Steps.
- `PROBLEMS/` — Real-world problems submitted by the community. Every
  problem has: ID, Description, Source, Status, Potential Impact.
- `VENTURES/` — Problems that became ventures. Document the complete
  journey.
- `BLUEPRINT/` — The evolving organizational blueprint. Defines:
  Purpose, Principles, Governance, Community Rules, Experiment Rules.
- `DIVINA/` — Everything related to Divina: Prompt, Memory,
  Capabilities, Roadmap, Architecture.
- `AGENTS/` — Specifications for every AI agent: Responsibilities,
  Inputs, Outputs, Tools, Constraints.
- `INTEGRATIONS/` — Telegram, Discord, WhatsApp, GitHub, MCP, A2A, etc.

# Contributor Journey

A contributor arrives → Reads README → Understands mission → Reads
START-HERE → Chooses one mission → Creates first Pull Request →
Receives recognition → Returns.

The journey should require less than 30 minutes.

---

# First Missions

1. Submit a problem.
2. Improve documentation.
3. Research an existing solution.
4. Design a better experiment.
5. Improve Divina.
6. Build an integration.
7. Interview someone affected by AI.
8. Translate documentation.

Every mission should be beginner-friendly.

---

# Divina

Divina is the AI steward of the experiment.

She does not govern. She does not decide. She helps.

Responsibilities: Welcome contributors. Collect problems. Recommend
issues. Summarize discussions. Generate documentation. Keep knowledge
organized. Match people with meaningful work.

---

# Community

The community is the product. The software enables the community.
Every decision should strengthen participation.

---

# Repository Design Principles

Simple. Optimistic. Welcoming. Transparent. Research-driven. Never
overwhelming.

Every page should answer: "What can I do next?"

---

# Coding Standards

Markdown-first. Specification-driven. Documentation before
implementation. Every feature begins with a specification. Every
experiment is documented. Every decision is explainable.

---

# GitHub Issues

Issues should be categorized: `good-first-issue`, `research`,
`documentation`, `design`, `community`, `agent`, `integration`, `bug`,
`experiment`, `help wanted`.

---

# GitHub Discussions

Discussions should become the primary place where ideas evolve before
implementation. The repository should encourage curiosity over
certainty.

---

# Long-Term Vision

365EXPT becomes the world's largest open laboratory studying how humans
and AI collaborate. Over time, the community develops: new
organizational patterns, new governance models, new AI agents, new
ventures, new forms of meaningful work.

All discoveries remain open source.

---

# Definition of Success

One year from now: thousands of contributors, hundreds of experiments,
dozens of ventures, new jobs that did not exist before — and a public
body of knowledge helping humanity understand how humans and AI can
build a better future together.

That is the purpose of this repository.
